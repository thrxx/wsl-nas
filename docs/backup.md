# 💾 Стратегии бэкапа и обслуживание WSL-NAS

> ⚠️ **Критическое предупреждение:** WSL2 засыпает вместе с Windows, не имеет аппаратного RAID и использует динамический виртуальный диск (`.vhdx`). Любая аппаратная неисправность, сбой питания Windows или повреждение VHDX приведёт к потере данных. **Регулярные бэкапы обязательны.**

## 📜 Правило 3-2-1 для WSL-NAS
| Принцип | Реализация в проекте |
|---------|----------------------|
| **3** копии данных | Оригинальная `/data` + локальный бэкап + облако/внешний диск |
| **2** разных носителя | SSD внутри ПК + внешний HDD/USB или NAS/Cloud |
| **1** копия вне дома | Облако (rclone, Backblaze, Yandex Disk) или оффсайт-сервер |

---

## 🛡️ Доступные стратегии бэкапа

| Метод | Как работает | Плюсы | Минусы | Надёжность |
|-------|--------------|-------|--------|------------|
| **📦 Снапшот VHDX** | Полная копия виртуального диска | Восстановление всей системы за минуты | Требует `wsl --shutdown`, большой размер | ⭐⭐⭐ |
| **🔄 `rsync` (инкрементальный)** | Синхронизация изменённых файлов | Быстро, прозрачно, сохраняет права | Занимает место на втором диске | ⭐⭐⭐⭐ |
| **🔐 `restic` / `borg`** | Дедупликация + шифрование | Экономия места, версионирование, защита от ransomware | Требует изучения, восстановление сложнее | ⭐⭐⭐⭐⭐ |
| **☁️ Cloud sync (`rclone`)** | Синхронизация с облаком | Оффсайт-хранение, доступ из любой точки | Зависит от интернета, лимиты провайдера | ⭐⭐⭐ |

## 📦 1. Снапшот VHDX (полный образ)
Самый простой метод для аварийного восстановления всей среды WSL.

### 📋 Порядок действий
```bash
# 1. Полностью остановите WSL (обязательно!)
wsl --shutdown

# 2. Найдите путь к виртуальному диску
wsl --manage Ubuntu --get-vhdx
# Пример: C:\Users\You\AppData\Local\Packages\Canonical...\LocalState\ext4.vhdx

# 3. Скопируйте файл на внешний носитель
xcopy /J /H "C:\Path\To\ext4.vhdx" "D:\Backups\wsl-nas\ext4_$(date +%F).vhdx"
```
🔒 **Важно:** Копирование `.vhdx` при работающем WSL приведёт к повреждённой копии. Всегда используйте `wsl --shutdown`.

## 🔄 2. Инкрементальный бэкап через `rsync`
Рекомендуемый метод для ежедневного сохранения файлов `/data`.

### 📜 Автоматизация (скрипт + cron)
```bash
# Сохраните как /usr/local/bin/nas-backup.sh
#!/usr/bin/env bash
set -euo pipefail

DEST="/mnt/d/nas-backup"  # Укажите путь к внешнему/второму диску
LOG="/var/log/nas-backup.log"

echo "[$(date)] 🚀 Starting rsync backup..." >> "$LOG"
rsync -avh --delete --info=progress2 /data/ "$DEST/" >> "$LOG" 2>&1
echo "[$(date)] ✅ Backup finished." >> "$LOG"
```
```bash
sudo chmod +x /usr/local/bin/nas-backup.sh
sudo crontab -e
# Добавьте строку (ежедневно в 02:00):
0 2 * * * /usr/local/bin/nas-backup.sh
```

### 🔄 Восстановление
```bash
rsync -avh --delete /mnt/d/nas-backup/ /data/
sudo chown -R $USER:$USER /data
```

## 🔐 3. Версионированный бэкап через `restic`
Защита от удаления, шифрования ransomware и повреждения метаданных.

### ⚙️ Инициализация репозитория
```bash
sudo apt install restic
# Локальный репозиторий
restic init --repo /mnt/e/restic-nas-repo
# Или удалённый (S3, B2, Rclone)
# restic init --repo rclone:backblaze:wslnas
```

### 📥 Создание бэкапа
```bash
restic -r /mnt/e/restic-nas-repo backup /data \
  --exclude="*.tmp" \
  --exclude=".cache" \
  --tag daily
```

### 🔍 Проверка и очистка
```bash
restic -r /mnt/e/restic-nas-repo snapshots
restic -r /mnt/e/restic-nas-repo check
# Удаление старых снапшотов (хранить 30 дней)
restic -r /mnt/e/restic-nas-repo forget --keep-within 30d --prune
```

### 🔄 Восстановление
```bash
restic -r /mnt/e/restic-nas-repo restore latest --target /data
```

## 📉 4. Обслуживание VHDX: сжатие виртуального диска
Динамический VHDX **растёт при записи, но не уменьшается автоматически** при удалении файлов.

### 🧹 Автоматическое сжатие (современный WSL)
```powershell
# Windows PowerShell (Admin)
wsl --shutdown
wsl --manage Ubuntu --compact
```
💡 Рекомендуется запускать раз в 2 недели или после массового удаления файлов.

### 🛠️ Альтернатива (Hyper-V модуль)
```powershell
# Если wsl --manage недоступен:
Optimize-VHD -Path "C:\Path\To\ext4.vhdx" -Mode Full
```

## 📅 Расписание обслуживания (чек-лист)

| Задача | Частота | Команда / Скрипт |
|--------|---------|------------------|
| Бэкап `rsync` | Ежедневно | `cron` → `nas-backup.sh` |
| Бэкап `restic` | 2–3 раза/нед | `restic backup /data` |
| Сжатие VHDX | Раз в 2 недели | `wsl --manage Ubuntu --compact` |
| Обновление ОС | Раз в неделю | `sudo apt update && sudo apt upgrade -y` |
| Проверка целостности | Раз в месяц | `restic check` / `testparm` / `health_check.sh` |
| Тестовое восстановление | Раз в квартал | Восстановить тестовую папку в `/tmp/restore_test` |

## 🚨 Что делать при аварии?

| Сценарий | Действия |
|----------|----------|
| **Удалены файлы по ошибке** | `restic restore latest --target /tmp/recovery` → выборочное копирование обратно |
| **Повреждён VHDX / WSL не запускается** | Удалить дистрибутив → `wsl --install` → скопировать чистый `.vhdx` из бэкапа → `wsl --import` |
| **Потеря прав доступа** | `sudo chown -R $USER:$USER /data` → `sudo find /data -type d -exec chmod 775 {} \;` |
| **Samba не видит данные** | Проверить `df -hT /data`, `systemctl restart smbd`, `testparm` |

## ✅ Чек-лист готовности бэкапов
- [ ] Настроен минимум 1 инкрементальный метод (`rsync` или `restic`)
- [ ] Есть оффсайт-копия (облако или физический диск вне дома)
- [ ] Скрипт бэкапа проверяет коды возврата и логирует ошибки
- [ ] Проведено **тестовое восстановление** хотя бы 1 раза
- [ ] VHDX сжат после удаления больших объёмов данных
- [ ] Пароли `restic`/`rclone` сохранены в менеджере паролей (не в скриптах!)
